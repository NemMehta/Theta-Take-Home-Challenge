# Design Notes

**Bundles are self-contained and schema-aligned, and they identify scored tests by node ID rather than by file.** A task bundle is a directory carrying everything an evaluation needs: a `task.json` of metadata (repo, pinned base commit, the pre-built image and its digest, the declared test runner, and the scored test files), the problem statement, the gold and test patches, and two manifests listing the fail-to-pass and pass-to-pass node IDs. The assignment's minimal example modeled the scored tests as separate `pass2pass/` and `fail2pass/` folders, but real SWE-bench Pro tasks identify their scored tests by pytest node ID inside files that also contain unrelated tests, so extracting them into standalone files isn't faithful. Storing node-ID manifests alongside the test patch keeps the bundle self-contained and lets the masker and scorer operate at the granularity the data actually uses.

**We pull pre-built per-instance images and pin them by digest, while keeping the container interface generic.** The slowness and brittleness the assignment calls out come from resolving each repository's dependency universe; SWE-bench Pro already solves this by shipping a working image per instance, so `task init` pulls `jefzda/sweap-images:<tag>` and records its sha256 digest into the bundle. That buys reproducibility and a fast demo without re-deriving environments. Crucially, nothing in the run logic hardcodes Python or pytest: `init` discovers where the repo lives inside the image, the test runner is declared in `task.json`, and scoring parses a standard report format — so extending to a build-from-Dockerfile flow for an arbitrary repo, or to a non-Python runner, is an additive change rather than a rewrite.

**The host orchestrates; the container executes.** All untrusted work — applying a solver's patch, running a repository's tests, running a command-style solver — happens only inside the container, while the host limits itself to issuing `docker` commands (we shell out to the CLI rather than use an SDK, so every command is transparent and logged for reproducibility and debugging) and reading results back. The run container is started with `--network none` and memory/CPU/PID limits, and we verify this rather than assert it: a solver probe confirms the network is unreachable. State lives in a stdlib SQLite database where each invocation records a command row and each run records aggregate counts plus a per-test results blob — and that blob is also the JSON report we emit as the evaluation artifact, so the queryable record and the deliverable are the same object.

**Hiding the scored tests turned out to require editing the baseline tree, not withholding the test patch — and that reframed the masker.** We initially assumed the scored tests are introduced by the test patch, so that simply not applying it would hide them. Reconnaissance disproved this: on our instance all nine scored tests already exist at the base commit (the fix is to source, and it flips the failing test to passing). Hiding therefore has to be an active edit of the baseline working tree. The simplest implementation deletes each scored test file — robust and language-agnostic, but it over-hides any unrelated test in the same file, conflicting with the requirement that the solver still see every non-scored test. The function-level masker parses each file with Python's `ast` and removes only the scored functions or methods, leaving unrelated tests intact; on our instance it preserves exactly the one unrelated test (`test_ansible_version`) that file-level hiding would have removed. The tradeoff is fidelity versus robustness: function-level is precise but Python-specific and more fragile, so it is the default for Python and falls back to a file-level delete whenever a scored test can't be located, a file won't parse, or an edit would produce invalid Python — the fail-safe always prefers over-hiding to the unacceptable failure of leaving a scored test visible.

**A run solves and then scores against a fresh baseline, which keeps masking and capture from contaminating the result.** Within one container, the solve phase reaches a clean baseline, masks the scored tests, and lets the solver produce a source patch; the score phase then resets to a fresh baseline (discarding the mask), applies that patch, re-stages the real test files, and runs the scored node IDs. Scoring on a pristine tree rather than the masked one guarantees the masker's edits can never leak into the score and makes scoring structurally identical to the gold-patch check in `validate`, which we had already proven correct. The same concern shapes patch capture: before diffing the solver's work we restore any masked files, so neither a deletion nor a function-level edit contaminates the recorded patch — verified by a command solver whose captured diff contains only its source edit.

**Scoring is deliberately version-stable.** The pre-built images ship old toolchains (pytest 6.1.2), so relying on modern conveniences like `pytest-json-report` would be brittle; instead we run with pytest's built-in JUnit XML output and parse that into a per-node outcome map, which is stable across pytest versions. One empirical discovery hardened this further: Ansible's CLI tests share a process-global singleton and fail order-dependently when run together in a single process, which would have silently corrupted every score. Running each test in its own subprocess (`pytest-forked`) makes results match the canonical baseline-vs-gold expectations exactly. We chose determinism over convenience throughout, and the same runner is shared by `validate` and `run` so the two can never disagree about what "passed" means.

**Reproducibility is built in via pinning and provenance.** Each bundle pins the base commit and the image digest; the instance commit used to stage the scored tests is derived from the instance id rather than hardcoded; the exact `docker` and `git` commands are issued explicitly and surfaced on failure; and every run emits one structured JSON report. Given the same bundle and image, the tool produces the same verdict on any machine — which is what lets a collaborator query a run id in the database and trust the recorded result instead of reproducing the run.
